# Security Policy

Arpass takes the security of stored vault data extremely seriously. This document explains how to report security issues, what we commit to in response, and what is in scope.

## TL;DR

- **Security contact**: security@arpass.io
- **Encrypted reports (PGP)**: see [PGP key](#pgp-key) below
- **GitHub Security Advisory**: https://github.com/technoblest/arpass-spec/security/advisories/new (preferred for code-level issues)
- **Initial response SLA**: 72 hours
- **Coordinated disclosure**: 90 days

## Reporting a vulnerability

We welcome reports from independent security researchers. Three reporting channels:

### Channel A — GitHub Security Advisory (preferred)

For code-level vulnerabilities (XSS, crypto misuse, auth bypass, etc.), open a private security advisory:

https://github.com/technoblest/arpass-spec/security/advisories/new

This creates a private discussion thread visible only to repo maintainers, with built-in CVE coordination.

### Channel B — Email (for sensitive disclosures)

For issues that require encrypted communication or that don't fit a GitHub advisory (e.g. infrastructure issues, third-party dependency vulnerabilities affecting Arpass deployments):

- **Address**: security@arpass.io
- **Encrypted email**: see [PGP key](#pgp-key)
- **Subject prefix**: `[Arpass Security]`

### Channel C — Bug Bounty (HackerOne)

A formal Bug Bounty program is published on HackerOne for in-scope reports with monetary rewards:

- **Program URL**: https://hackerone.com/arpass *(planned, will be updated when live)*
- **Reward range**: see program page (typical: USD 100 – 5,000 depending on severity)

## Build verification

Arpass is a browser app, which means **every visit downloads JavaScript that runs with full access to your master password and Recovery Secret while you're on the page**. The integrity of that JavaScript is therefore a security boundary.

To make tampering detectable, every commit to `main` records the SHA-256 of every file under `web/` (the directory that Cloudflare Pages serves) in two places:

- [`BUILD-HASHES.md`](./BUILD-HASHES.md) — human-readable table at repo root
- [`web/.well-known/build-hashes.txt`](https://arpass.io/.well-known/build-hashes.txt) — plain-text file served at `https://arpass.io/.well-known/build-hashes.txt`

These hashes are generated by [`.github/workflows/build-hashes.yml`](./.github/workflows/build-hashes.yml) on every push to `main` (the workflow runs `node scripts/generate-build-hashes.mjs` and commits the result back).

### How to verify a deployed file

```bash
# 1. Fetch the live file from arpass.io
curl -s https://arpass.io/lib/vault-crypto.js | sha256sum
# Example output:
#   3a4f...  -

# 2. Find the same path in BUILD-HASHES.md (or build-hashes.txt) at the
#    commit currently deployed. The footer of the LP shows the build commit.
grep "lib/vault-crypto.js" BUILD-HASHES.md

# 3. Hashes must match. If they diverge, the live deployment is shipping
#    something different from what is in this repo at this commit.
```

A divergence is **not necessarily malicious** (deploy lag, CDN edge variance, etc.), but it is something to investigate. Open a [security advisory](https://github.com/technoblest/arpass-spec/security/advisories/new) if confirmed.

### What this does NOT protect against

- A compromise of the GitHub repository itself (the hashes are committed by GitHub Actions, so an attacker with repo write access can rewrite both the code and the hashes in the same commit).
- A first-visit attack where the attacker has held the bundle modified for less than the time between deploy and your visit, then reverted (window-of-vulnerability attack).

For the strongest protection, install a future browser-extension version (planned, see roadmap), where the bundle is signed and updated through the Web Store review process.

## Scope

### In scope (we will fix and credit)

- **Cryptographic implementation bugs** in `web/lib/vault-crypto.js`, `web/lib/vault-client.js`, `web/lib/client-auth.js`, and the corresponding server `functions/_lib/auth.js`
- **Envelope format vulnerabilities** (v2 / v3 / v4 / v4.1 — see `docs/crypto-2of3.md`)
- **Authentication/authorization bugs** in `functions/api/*` endpoints
- **Cross-site scripting (XSS)** in `web/index.html`, `web/app.html`, or any served resource
- **CSRF** affecting state-changing endpoints
- **Server-side request forgery (SSRF)** via the Cloudflare Pages Functions
- **Injection vulnerabilities** (SQL/NoSQL/command/HTML/template) in any code path
- **Dependency vulnerabilities** that materially affect Arpass deployment
- **Misconfiguration** of HTTP security headers (`web/_headers`), CSP, HSTS, etc.
- **Privacy leaks** through Arweave tags, GraphQL queries, or response metadata
- **Stripe webhook signature verification** flaws
- **Identity migration / key rotation** flaws (`/api/vault/migrate`)
- **Side-channel attacks** that are practically exploitable (timing, memory)

### Out of scope

- Self-XSS that requires the victim to paste attacker-controlled content into DevTools
- DoS via raw resource exhaustion (e.g. opening 10,000 tabs)
- Bugs in Arweave gateway, Cloudflare CDN, Stripe, or other third-party infrastructure (please report directly to those vendors; we are happy to coordinate if needed)
- Social engineering of Technoblest staff or users
- Physical attacks on Technoblest infrastructure
- Brute-force attacks on weak Master Passwords (this is a fundamental property of password-based auth; we mitigate via PBKDF2-600K + mandatory Passkey but we cannot prevent users choosing weak passwords)
- Vulnerabilities in browsers, OSes, or hardware authenticators (please report to the vendor; we will support you in coordinating)
- Reports based on automated scanner output without a working exploit demonstration
- Theoretical issues without a practical exploitation path
- Spam, social media, or unrelated infrastructure issues

## Safe harbor

We will not pursue legal action against researchers who:

1. Make a good-faith effort to avoid privacy violations, destruction of data, or service degradation during research
2. Only test against accounts they own or have explicit permission to test
3. Do not exploit a vulnerability beyond what is necessary to demonstrate it
4. Report the issue promptly via the channels listed above
5. Give us a reasonable time (per the SLA above) to remediate before public disclosure

If you act in good faith and follow this policy, we consider your activities authorized under the Japanese Unauthorized Computer Access Law and equivalent international law.

## What is NOT a security guarantee

Arpass uses 2-of-3 client-side encryption (AES-256-GCM + PBKDF2-SHA256-600K + HKDF + Passkey-PRF) backed by Arweave permanent storage. We commit to **best-effort security** consistent with industry practice, but **we do NOT and CANNOT guarantee absolute safety**:

- **No cryptosystem is provably unbreakable** in finite time. PBKDF2 + AES-256-GCM is industry-standard but may be weakened by future research, quantum computers, or implementation bugs (including ours).
- **Arweave is publicly readable**. Anyone can download your encrypted envelope. Confidentiality depends entirely on the encryption being strong AND your factors (Master Password / Passkey / Recovery Secret) remaining secret.
- **Sufficiently weak Master Passwords are GPU-attackable** even with PBKDF2. We mitigate this by mandating Passkey + WebAuthn PRF as a second factor (v4.1 onwards), but the password itself remains a meaningful entropy source.
- **Users who lose 2 of the 3 recovery factors permanently lose access**. This is intentional zero-knowledge design, not a bug to report.
- **Past Arweave writes carry their own metadata**. Tags written before v4.1 (2026-04-29) included a plaintext `vault-id` tag — see `docs/security-baseline.md §6-6` for the disclosure. Future writes do not.

For high-stakes accounts (banking primary, government, primary email-account), users are strongly encouraged to use Arpass *as one of multiple* password managers, not as a single point of failure.

## What we publish after a fix

For every accepted security report, we publish:

1. A GitHub Security Advisory with CVE assignment (when applicable)
2. A patch commit referencing the advisory
3. An entry in `docs/security-baseline.md` if the issue reveals a class of bugs needing systemic prevention (e.g. a new grep test in `scripts/lint-security.sh`)
4. Public credit to the reporter (unless anonymity is requested)
5. For critical issues: a blog post on https://arpass.io/blog (planned)
